# ER Diagram Description

## Overview

The ER Diagram represents the logical database design of the **Personalized Networking Assistant** project. Although the application currently stores its data in local JSON files (`history.json` and `feedback.json`), the system follows a relational database structure. The diagram illustrates how conversation sessions, generated topics, talking points, and user feedback are logically connected.

---

# Entities

## 1. ConversationSession

This entity stores the information of every networking conversation session generated by the system. Each session is uniquely identified by a Conversation ID and contains the event details provided by the user.

### Attributes
- **conversation_id (PK)** – Unique identifier of the conversation session.
- **event_name** – Name of the networking event.
- **timestamp** – Time when the conversation was generated.
- **logged_at** – Time when the record was stored.

---

## 2. Topic

This entity stores all conversation topics generated for a particular networking session. A single conversation session can produce multiple topics.

### Attributes
- **topic_id (PK)** – Unique identifier of the topic.
- **conversation_id (FK)** – References ConversationSession.
- **topic** – Generated networking topic.
- **fact_check_status** – Indicates whether the topic has been verified.

---

## 3. TalkingPoint

This entity stores detailed talking points generated for each topic. Every topic may contain multiple talking points that help users start meaningful conversations.

### Attributes
- **point_id (PK)** – Unique identifier of the talking point.
- **topic_id (FK)** – References Topic.
- **point** – Main conversation point.
- **details** – Additional explanation of the talking point.

---

## 4. Feedback

This entity stores ratings and comments submitted by users after using the conversation assistant.

### Attributes
- **feedback_id (PK)** – Unique identifier of feedback.
- **conversation_id (FK)** – References ConversationSession.
- **rating** – User rating (1–5).
- **comments** – User comments or suggestions.
- **logged_at** – Time when feedback was recorded.

---

# Relationships

### ConversationSession → Topic
**Relationship:** One-to-Many (1:N)

One conversation session can generate multiple networking topics, while each topic belongs to only one conversation session.

---

### Topic → TalkingPoint
**Relationship:** One-to-Many (1:N)

Each generated topic can contain multiple talking points, whereas every talking point belongs to one topic only.

---

### ConversationSession → Feedback
**Relationship:** One-to-One (1:1)

Each conversation session can receive one feedback record from the user.

---

# Primary Keys

- ConversationSession → **conversation_id**
- Topic → **topic_id**
- TalkingPoint → **point_id**
- Feedback → **feedback_id**

---

# Foreign Keys

- Topic → **conversation_id**
- TalkingPoint → **topic_id**
- Feedback → **conversation_id**

---

# Constraints

- Every **conversation_id** must be unique.
- Every **topic_id** must reference a valid ConversationSession.
- Every **point_id** must reference a valid Topic.
- Every **feedback** must reference a valid ConversationSession.
- Rating values are restricted between **1 and 5**.
- A talking point cannot exist without a corresponding topic.

---

# Normalization

The database design follows normalization principles by separating conversation sessions, generated topics, talking points, and user feedback into independent entities. This minimizes redundancy, improves data consistency, and maintains referential integrity.
